## 一、这是什么

**UCC（Unified Claude Code Config）** 是一套 Claude Code 配置插件，部署后能让 Claude Code 具备：

- 一致的编码规范（Go / TypeScript / Vue / JavaScript / Node.js）
- 专业代理（Agent）处理特定任务
- 斜杠命令触发标准化工作流
- 可选的 Hooks 自动拦截危险操作

## 二、如何部署

### 方式 A：全局部署（推荐个人使用）

将本仓库内容放到 Claude Code 全局配置目录，对所有项目生效：

```bash
# 克隆到 Claude Code 全局目录
git clone <repo-url> ~/.claude/

# 或者只拷贝关键目录
cp -r agents/ commands/ rules/ skills/ contexts/ ~/.claude/
cp CLAUDE.md ~/.claude/CLAUDE.md
```

### 方式 B：项目级部署（推荐团队使用）

将本仓库作为子模块或直接复制到项目根目录下的 `.claude/` 目录：

```bash
# 在你的项目根目录，以子模块方式引入
git submodule add <repo-url> .claude
```

### 方式 C：当前仓库直接使用（已部署）

本仓库自身就是配置集，在此目录下打开 Claude Code 即可使用。

## 三、命名空间与验证

每次对话中，可通过两种方式确认「UCC 配置已生效」：

```
# 方式1：在消息中加 @ucc 标记
@ucc 帮我写一个 Go HTTP handler

# 方式2：使用 /ucc-* 斜杠命令
/ucc-plan 实现用户登录功能
```

**生效标志**：输出末尾出现 `配置标识：UCC`

如果没有出现，说明 Claude Code 没有加载到本配置，检查 `CLAUDE.md` 是否在正确目录。

## 五、代理（Agent）使用说明

代理是专业化的子任务处理器，有两种触发方式。

### 5.1 通过斜杠命令隐式触发（推荐）

```bash
/ucc-plan        → 自动调用 planner 代理
/ucc-tdd         → 自动调用 tdd-guide 代理
/ucc-go-review   → 自动调用 go-reviewer 代理
```

### 5.2 直接在对话中请求

```
帮我用 planner 代理规划这个功能
用 security-reviewer 检查这段代码
```

### 5.3 代理自动触发规则（Claude 会主动调用）

| 触发条件 | 自动调用的代理 |
|---------|---------------|
| 代码写完 | `code-reviewer` 或语言专项 reviewer |
| 含敏感操作的代码 | `security-reviewer` |
| 构建/类型检查失败 | `build-error-resolver` |
| 文档/接口变更 | `doc-updater` |
| 复杂功能请求 | `planner` |
| 数据库变更 | `database-reviewer` |

## 六、工作模式（Contexts）说明

三种模式影响 Claude 的行为风格：

| 模式 | 行为特点 | 适用场景 |
|------|---------|----------|
| **开发模式** (`dev`) | 先写代码，后解释；优先可工作方案 | 活跃编码期 |
| **审查模式** (`review`) | 先读懂代码，再评论；输出发现报告 | PR 审查 |
| **研究模式** (`research`) | 先理解，后行动；输出调研结论 | 探索新技术 |

切换命令：`/ucc-context-dev` / `/ucc-context-review` / `/ucc-context-research`

## 七、Hooks 激活说明（重要）

Hooks 是**硬性拦截**机制，默认未激活。需要手动合并到 `~/.claude/settings.json`。

### 7.1 三个 Hook 的作用

| Hook | 触发时机 | 拦截内容 |
|------|---------|----------|
| `pre:cmd:risk-blocker` | AI 执行 Shell 命令前 | `rm -rf`、危险格式化命令 |
| `pre:write:sensitive-check` | AI 写文件前 | 硬编码密码、API Key、Token |
| `post:write:ts-check` | 写入 TS/TSX/Vue 文件后 | 自动运行 `tsc --noEmit` + ESLint |
| `stop:delivery-reminder` | 每次响应完成后 | 输出交付检查提醒（不阻断）|

### 7.2 激活方式

Hooks 脚本使用**相对路径**调用，路径相对于 Claude Code 启动时的工作目录解析。因此激活方式取决于你的部署场景。

#### 场景 A：仅在本仓库目录下使用（最简单）

直接将 `hooks/hooks.json` 的 `hooks` 字段复制到本仓库的 `.claude/settings.json`（或全局 `~/.claude/settings.json`），在 `agent-config/` 目录下打开 Claude Code 时即生效：

```json
// .claude/settings.json 或 ~/.claude/settings.json
{
  "hooks": {
    "PreToolUse": [ /* 从 hooks/hooks.json 复制 */ ],
    "PostToolUse": [ /* 从 hooks/hooks.json 复制 */ ],
    "Stop": [ /* 从 hooks/hooks.json 复制 */ ]
  }
}
```

#### 场景 B：在任意项目目录下全局生效

hooks 命令中的 `scripts/hooks/...` 是相对路径，在其他项目目录下会找不到脚本。需要将路径替换为**绝对路径**后再写入 `~/.claude/settings.json`：

```json
// 将相对路径替换为绝对路径，例如：
// 原来："node scripts/hooks/run-with-flags.js pre:cmd:risk-blocker ..."
// 改为："node /绝对路径/agent-config/scripts/hooks/run-with-flags.js pre:cmd:risk-blocker ..."
{
  "hooks": {
    "PreToolUse": [
      {
        "matcher": "Bash|PowerShell|Cmd",
        "hooks": [{
          "type": "command",
          "command": "node /your/path/to/agent-config/scripts/hooks/run-with-flags.js pre:cmd:risk-blocker scripts/hooks/pretool-risk-blocker.js standard,strict"
        }]
      }
    ]
  }
}
```

> **注意**：Hooks 依赖 `scripts/hooks/` 下的 Node.js 脚本，激活前确保已安装 Node.js。

## 八、规则体系说明

`rules/` 目录是 Claude 编码行为的约束核心：

```
rules/
├── common/        # 通用规则（所有语言适用）
│   ├── security   # OWASP Top 10 安全检查
│   ├── git        # Conventional Commits 规范
│   ├── tdd        # 测试驱动开发规则
│   └── ...（共9个文件）
├── golang/        # Go 专项：错误处理、并发、惯用法（5个文件）
├── typescript/    # TS/Vue 专项：类型安全、组合式 API（5个文件）
└── javascript/    # JS 专项：异步模式、闭包、XSS 防护（5个文件）
```

修改规则直接编辑对应文件即可，Claude Code 每次启动时自动加载。

## 九、典型工作流示例

### 开发新功能

```
1. /ucc-plan 用户注册功能需求...
   → planner 代理输出实现方案

2. /ucc-tdd
   → tdd-guide 代理引导先写测试

3. 编写代码（Claude 自动调用 code-reviewer）

4. /ucc-quality-gate
   → 提交前综合检查

5. git commit（Conventional Commits 格式）
```

### 修复 Go 构建错误

```
1. 构建失败时直接发送错误信息
   → Claude 自动调用 go-build-resolver

2. 或者使用 /ucc-go-build
```

### 团队 PR 审查

```
1. /ucc-context-review    # 切换到审查模式
2. /ucc-code-review       # 通用审查
3. /ucc-go-review         # 或语言专项审查
```

## 十、配置定制入口

详细定制方法参见 `docs/配置定制指南.md`，以下是常见定制场景：

### 10.1 添加新的编码规则

在对应语言目录下新建 `.md` 文件：

```bash
# Go 新规则
echo '# 自定义规则\n...' > rules/golang/my-rule.md

# 通用规则
echo '# 自定义规则\n...' > rules/common/my-rule.md
```

### 10.2 添加新的斜杠命令

在 `commands/` 下新建 `.md` 文件，参考现有命令格式：

```yaml
---
name: ucc-my-command
description: 我的自定义命令
---

# 命令内容...
```

### 10.3 在项目中覆盖全局规则

在项目的 `.claude/CLAUDE.md` 中追加覆盖说明，项目级规则优先级高于全局。

### 10.4 关闭某个 Hook

编辑 `~/.claude/settings.json`，删除对应的 hook 条目即可。
