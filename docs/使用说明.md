## 一、这是什么

**UCC（Unified Claude Code Config）** 是一套 Claude Code 配置插件，部署后可以提供：

- 一致的编码规范（Go / TypeScript / Vue / JavaScript / Node.js）
- 专业代理（Agent）处理特定任务
- 显式完整命令触发标准化工作流
- 自动推进 runtime，在需要时再暂停人工介入
- 可选 Hooks 拦截危险操作

核心思想不是“给很多阶段命令”，而是“用少量可读入口启动完整自动化流程”。

## 二、如何部署

### 方式 A：项目级部署（推荐）

默认推荐的安全安装只复制 `.claude/` 下的运行时资产，不默认接管目标项目的根目录 `CLAUDE.md` 和 `.claude/settings.json`。

推荐使用复制脚本：

```bash
# 默认选择式安装：运行后按提示选择 CLAUDE、hooks、MCP 和平台
node scripts/copy-config.js /path/to/your-project

# 高级覆盖：显式生成 .claude/CLAUDE.md
node scripts/copy-config.js /path/to/your-project --claude-mode dotclaude

# 高级覆盖：生成根目录 bootstrap CLAUDE.md，并启用项目级 hooks
node scripts/copy-config.js /path/to/your-project --claude-mode import-root --hooks project

# 仅启用本地 hooks
node scripts/copy-config.js /path/to/your-project --hooks local

# 非交互生成项目级 MCP（默认 basic 预设）
node scripts/copy-config.js /path/to/your-project --mcp project

# 兼容旧布局
node scripts/copy-config.js /path/to/your-project --legacy-layout
```

默认复制完成后，目标项目结构应为：

```text
your-project/
└── .claude/
    ├── agents/
    ├── commands/
    ├── skills/
    ├── scripts/
    ├── workflows/
    └── README.md
```

脚本默认会进入选择式向导；只有在向导中选择或显式传入 `--claude-mode`、`--hooks` 或 `--mcp` 时，才会额外生成 `CLAUDE.md`、`.claude/CLAUDE.md`、`.claude/settings.json`、`.claude/settings.local.json` 或项目根 `.mcp.json`。

交互式模式会优先自动识别平台，并通过预设减少输入：

- `basic`：默认启用 `memory`、`filesystem`、`context7`
- `research`：在 `basic` 基础上额外启用 `sequential-thinking`
- `full`：启用全部内置 MCP，并提示输入所需 Key
- `custom`：手动选择 MCP 服务

只有 API Key、目录等必填参数需要输入；例如 `filesystem` 默认会使用目标项目根目录，直接回车即可确认。

### 方式 B：手动复制

```bash
cp -r README.md rules agents commands skills hooks scripts mcp-configs workflows /path/to/your-project/.claude/

# 可选：显式放置 .claude/CLAUDE.md 或 .claude/CLAUDE.ucc.md
# 可选：将 hooks 片段合并到 .claude/settings.json 或 .claude/settings.local.json
```

默认部署包不会复制 `docs/`、`tests/` 和 `contexts/`；这些目录只保留在 UCC 源仓库中用于维护和回归验证。

## 三、命名空间与验证

推荐直接用显式完整入口：

```text
/ucc-team-standard 审查当前项目框架并完成必要改造、验证、文档同步和总结
/ucc-single-standard 基于当前仓库完成自动检查、修改、验证和总结
@ucc 帮我分析这段 TypeScript 代码
```

**生效标志**：输出末尾出现 `配置标识：UCC`

如果没有出现，优先检查：

1. `.claude/commands/` 和 `.claude/agents/` 是否完整复制
2. `.claude/workflows/definitions.json` 是否已复制
3. 如果使用了 `--claude-mode dotclaude`，确认 `.claude/CLAUDE.md` 已生成；如果使用了 `--claude-mode import-root`，确认根目录 `CLAUDE.md` 或导入片段已到位
4. 如果启用了 hooks，确认 `.claude/settings.json` 或 `.claude/settings.local.json` 已成功生成或合并
5. 如果启用了项目级 MCP，确认项目根 `.mcp.json` 已生成，且 `.claude/settings.json` 中包含 `enabledMcpjsonServers`

如果你在维护 UCC 源仓库，运行 `node scripts/validate-config.js` 时，当前校验也会覆盖 workflow 语义闭环、双 hook 配置中的 `PostToolUse`，以及 docs / hooks / rules / skills 中是否残留退役 `/ucc-*` 命令。

## 四、公开命令面（8个）

### 4.1 自动化流程入口

| 命令 | 作用 | 自动链路 |
|------|------|----------|
| `/ucc-team-standard` | 标准团队交付闭环 | 澄清 -> 计划（节点内并行） -> 实施 -> 审查（节点内并行） -> 验证（节点内有限并行） -> 文档 -> 总结 |
| `/ucc-team-strict` | 严格团队交付闭环 | 澄清 -> 风险 -> 详细计划（节点内并行） -> 实施 -> 审查（节点内并行） -> 完整验证（节点内有限并行） -> 文档 -> 质量门禁 -> 总结 |
| `/ucc-team-research` | 团队调研闭环 | 问题定义 -> 证据 -> 结论 -> 交接 -> 自动切入 `team.standard.plan` |
| `/ucc-single-standard` | 单 agent 标准闭环 | 澄清 -> 计划 -> 实施 -> 审查 -> 验证 -> 总结 |
| `/ucc-single-research` | 单 agent 调研闭环 | 问题定义 -> 证据 -> 结论 -> 后续动作 -> 自动切入 `single.standard.plan` |

### 4.2 流程控制命令

| 命令 | 作用 |
|------|------|
| `/ucc-flow-status` | 查看当前或指定 workflow run 的状态，以及最近阶段摘要、并行委派状态、验证状态 |
| `/ucc-flow-continue` | 继续当前或指定的 paused run |
| `/ucc-flow-abort` | 中止当前或指定的 workflow run |

### 4.3 内部能力说明

构建修复、Go / TypeScript / JavaScript 专项审查、数据库审查、E2E、设计文档、交付文档、覆盖率补齐、死代码清理和 context 切换能力仍然存在，但它们已经内收为内部 agent 或 workflow 节点，不再作为用户直接输入的公开 slash 命令。

## 五、B 方案自动化流程

这部分回答一个核心问题：**用户触发命令后，会不会按照需求自动走完全流程？**

答案是：**会**。前提是使用的是上面 5 个显式 flow 入口之一。

### 5.1 `/ucc-team-standard`

适用场景：

- 已有项目框架雏形
- 希望自动检查、提出修改、实施修改、验证、收尾、更新文档
- 希望尽量少人工干预

输入方式：

```text
/ucc-team-standard 审查当前项目框架，识别结构问题并给出最小可执行修改；完成修改、验证、收尾并同步文档
```

系统预期流程：

1. `clarify`：澄清目标、边界、限制
2. `plan`：输出最小可执行计划
3. `implement`：实施改动
4. `review`：检查风险、回归和遗漏
5. `verify`：执行验证（节点内有限并行）
6. `docs`：同步必要文档
7. `summary`：输出交付总结

补充说明：`plan` 节点会并行调度 `planner`，并在命中架构类风险信号时按需调度 `architect`；`review` 节点会并行调度审查代理；`verify` 节点在命中 `db-migration` 时会按需并行调度 `database-reviewer`。

用户在正常情况下不需要继续输入下一阶段命令。

### 5.2 `/ucc-team-strict`

适用场景：

- 核心模块
- 高风险改动
- 需要完整质量门禁

输入方式：

```text
/ucc-team-strict 审查并重构权限、配置加载与发布链路，要求完整验证、质量门禁和文档同步
```

系统预期流程：

1. `clarify`
2. `risk`
3. `detailed-plan`
4. `implement`
5. `review`
6. `full-verify`（节点内有限并行）
7. `docs`
8. `quality-gate`
9. `summary`

补充说明：`detailed-plan` 节点会并行调度 `planner`，并在命中架构类风险信号时按需调度 `architect`；`review` 节点会并行调度审查代理；`full-verify` 节点在命中 `db-migration` 时会按需并行调度 `database-reviewer`。

### 5.3 `/ucc-team-research`

适用场景：

- 先研究现状
- 需要证据支持的改造建议
- 希望研究结束后自动衔接实施

输入方式：

```text
/ucc-team-research 评估当前项目框架的模块划分、构建结构和测试组织；若结论明确，则继续落地改造
```

系统预期流程：

1. `define-problem`
2. `evidence`
3. `conclusion`
4. `handoff`
5. 自动切入 `team.standard.plan`
6. 继续执行实施、审查、验证、文档同步和总结

这就是 team 模式下“研究后自动接力到交付”的标准套路。

### 5.4 `/ucc-single-standard`

适用场景：

- 单人维护
- 日常修复
- 中小型重构

输入方式：

```text
/ucc-single-standard 基于当前仓库完成自动检查、必要修改、验证和总结
```

系统预期流程：

1. `clarify`
2. `plan`
3. `implement`
4. `review`
5. `verify`
6. `summary`

这一条 single 标准链路已经完整定义完成；正常情况下不需要再人工补阶段命令。

### 5.5 `/ucc-single-research`

适用场景：

- 单人调研
- 需要先分析问题和证据
- 希望在可执行时自动继续落地

输入方式：

```text
/ucc-single-research 分析当前仓库的模块边界、构建组织和测试缺口，并在结论明确后继续落地
```

系统预期流程：

1. `define-problem`
2. `evidence`
3. `conclusion`
4. `next-action`
5. 自动切入 `single.standard.plan`
6. 继续执行实施、审查、验证和总结

也就是说，单人调研链路在结论明确后会直接接上单人标准闭环，而不是停在研究总结阶段。

## 六、何时需要继续输入命令

判断规则可以简化成一句话：

**只要流程还是 `running`，就不要继续手工输入下一阶段命令。**

### 6.1 不需要继续输入的情况

1. 刚刚启动了 `/ucc-team-*` 或 `/ucc-single-*`
2. 输出里明确说明会自动推进下一节点
3. 当前状态仍是 `running`

### 6.2 需要继续输入的情况

只在以下情况才需要人工接力：

1. 状态变成 `paused`
2. 输出里明确要求补充信息
3. 验证失败，需要确认处理方向
4. 存在高风险动作被 pausePolicy 拦住

此时统一使用：

```text
/ucc-flow-status
/ucc-flow-continue
```

`/ucc-flow-status` 除了基础 run 信息，还会优先展示 `.claude/workflows/control/latest.json` 中的最近阶段摘要、并行委派状态和验证状态，便于恢复上下文与判断阻塞点。

如果需要终止：

```text
/ucc-flow-abort
```

## 七、内部能力与公开边界

当前公开给团队成员的命令只有 8 个。团队培训时，只需要讲这 8 个：

1. `/ucc-team-standard`
2. `/ucc-team-strict`
3. `/ucc-team-research`
4. `/ucc-single-standard`
5. `/ucc-single-research`
6. `/ucc-flow-status`
7. `/ucc-flow-continue`
8. `/ucc-flow-abort`

其余能力已经转为内部自动调度，不再要求用户记住额外命令。例如：

- 构建或类型检查失败时，runtime 可自动切入构建修复链路
- 需要数据库审查时，runtime 可自动调度 `database-reviewer`
- 需要文档同步时，runtime 可自动调度 `doc-updater`、`design-doc-writer` 或 `delivery-doc-writer`
- 需要关键流程验证时，runtime 可自动调度 `e2e-runner`；而在 `verify` / `full-verify` 节点命中 `db-migration` 时，可按需并行调度 `database-reviewer`
- 需要语言专项审查时，runtime 可自动调度对应 reviewer agent

所有自定义 agent 默认 `model: inherit`，继承当前会话模型，不额外绑定固定官方模型别名。

## 八、生产环境建议

### 8.1 默认推荐

- 常规任务：`/ucc-team-standard`
- 高风险任务：`/ucc-team-strict`
- 先研究再落地：`/ucc-team-research`
- 单人闭环：`/ucc-single-standard`

### 8.2 不建议的做法

- 不要把每个阶段再拆成用户手工命令
- 不要让团队成员去记 profile、node、pausePolicy 参数
- 不要重新暴露低频专项 slash 命令，除非它已经成为高频主入口
- 不要公开推荐已经移除的旧入口

### 8.3 配置修改后的最小验证

维护 UCC 配置仓库时：

```bash
node scripts/validate-config.js
node tests/run-all.js
```

在业务项目中验证已部署的 UCC 资产时：

```bash
node .claude/scripts/validate-config.js
```

如果你的目标是“让团队成员一眼看懂，并且一条命令就能在生产场景中走完整自动化流程”，当前这套 B 方案和 8 个公开入口就是为这个目标设计的。
