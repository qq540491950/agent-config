# UCC 配置定制指南

> 面向生产场景的显式命令面与自动接力 workflow 定制手册

**文档版本：** v4.0.0

---

## 1. 设计目标

这套配置当前的核心目标不是“给很多零散命令”，而是：

1. 让用户通过极少数显式完整入口就能启动完整流程
2. 让 runtime 在 team / single 两种模式下自动接力执行
3. 将人工介入压缩到暂停、失败、冲突和收尾确认这些必要时刻
4. 在生产场景下仍能保留风险控制、状态追踪和恢复能力

如果你的团队已经有项目框架雏形，最常见的诉求通常不是“下一步还要输什么命令”，而是：

- 启动一次流程
- 自动检查
- 提出修改
- 实施修改
- 验证
- 收尾
- 更新文档

UCC v4 的命令面和 runtime 就是围绕这个目标收敛的。

## 2. 当前公开命令面

### 2.1 目录快照

```text
agents/                 # 代理配置（20个）
commands/               # 斜杠命令（38个）
contexts/               # 工作模式（3个）
skills/                 # 技能模块（19个）
workflows/              # workflow 定义与运行时状态
legacy/commands/        # 历史入口归档
```

### 2.2 为什么改成显式完整命令

旧模式的问题通常是：

1. 用户需要记住参数化入口和阶段性入口
2. 看不出当前到底是 team 还是 single
3. 很多人会误以为还要继续手工输入下一阶段命令
4. 命令数量一多，团队成员容易误用

改成显式完整命令后：

- `/ucc-flow-team-standard` 一眼就知道是团队标准流程
- `/ucc-flow-single-dev` 一眼就知道是单人开发闭环
- `/ucc-flow-continue` 一眼就知道是继续暂停流程

这更适合团队协作和生产使用。

### 2.3 公开命令分层

#### A. 显式流程入口

- `/ucc-flow-team-standard`
- `/ucc-flow-team-fast`
- `/ucc-flow-team-strict`
- `/ucc-flow-team-review`
- `/ucc-flow-team-research`
- `/ucc-flow-team-doc`
- `/ucc-flow-single-dev`
- `/ucc-flow-single-review`
- `/ucc-flow-single-research`

#### B. 流程控制

- `/ucc-flow-status`
- `/ucc-flow-continue`
- `/ucc-flow-abort`

#### C. 专项能力

- 语言审查、构建修复、文档生成、E2E、测试覆盖率、会话管理等命令继续保留
- 这些命令不是主入口，而是补充能力

推荐的团队认知模型是：

1. 日常交付，先选一个 `/ucc-flow-team-*`
2. 单人闭环，先选一个 `/ucc-flow-single-*`
3. 流程暂停后，只用 `/ucc-flow-status` 和 `/ucc-flow-continue`
4. 只有在需要专项深挖时，才额外使用专项能力命令

## 3. 自动接力运行模型

### 3.1 统一规则

所有公开 flow 入口都满足：

- `workflowCapable: true`
- `executionMode: auto`
- `triggerVisibility: always`

也就是说，用户一旦触发入口，runtime 默认就会自动推进，而不是每到一阶段都要求人工重新下命令。

### 3.2 暂停策略

当前 runtime 使用 `pausePolicy` 控制是否暂停：

- `auto`：最少打断，偏向小步自动闭环
- `balanced`：默认平衡，适合常规生产场景
- `strict`：更积极暂停，适合高风险改动

运行时只在这些情况下暂停：

1. 命中对应 `pausePolicy` 的风险信号
2. 关键执行失败
3. 上下文冲突
4. 缺少不能合理推断的关键输入

其他情况下默认继续自动推进。

### 3.3 为什么统一成 continue

无论 team 还是 single，都统一使用：

```text
/ucc-flow-continue
```

这样做有三个好处：

1. 团队成员不用记多套“恢复/接着跑”命令
2. 文档和培训成本更低
3. 自动化系统也更容易封装固定入口

## 4. 你当前场景的推荐用法

你的场景是：

- 项目框架已经有雏形
- 希望自动化检查、提出修改、修改、验证、收尾、更新文档
- 希望 team 模式或单 agent 模式都能走完整闭环
- 希望人工干预尽量少

这对应的推荐方式如下。

### 4.1 团队全流程自动治理

直接输入：

```text
/ucc-flow-team-standard 审查当前项目框架，识别结构问题并给出最小可执行修改；完成修改、验证、收尾并同步文档
```

系统预期行为：

1. `clarify`
2. `plan`
3. `implement`
4. `review`
5. `verify`
6. `docs`
7. `summary`

你不需要继续手工输入“计划命令、开发命令、验证命令、文档命令”。
只有在流程暂停时，才需要：

```text
/ucc-flow-status
/ucc-flow-continue
```

### 4.2 团队快速闭环

适合改动小、风险低：

```text
/ucc-flow-team-fast 修复当前框架中的构建与目录边界问题，并完成最小验证
```

预期自动链路：

1. `clarify`
2. `mini-plan`
3. `implement`
4. `mini-verify`
5. `summary`

### 4.3 团队严格闭环

适合核心模块、高风险改动：

```text
/ucc-flow-team-strict 审查并重构权限、配置加载与发布链路，要求完整验证、质量门禁和文档同步
```

预期自动链路：

1. `clarify`
2. `risk`
3. `detailed-plan`
4. `implement`
5. `review`
6. `full-verify`
7. `docs`
8. `quality-gate`
9. `summary`

### 4.4 先研究，再自动交接实施

如果你不确定该怎么改，但希望研究完自动衔接到交付：

```text
/ucc-flow-team-research 评估当前项目框架的模块划分、构建结构和测试组织；若结论明确，则继续落地改造
```

预期自动链路：

1. `define-problem`
2. `evidence`
3. `conclusion`
4. `handoff`
5. 自动跳转到 `team.standard.plan`
6. 继续执行实施、审查、验证、文档和收尾

这就是“研究后自动接力”的推荐套路。

### 4.5 单 agent 自动闭环

如果不需要 team 编排，而是希望单人闭环：

```text
/ucc-flow-single-dev 审查当前仓库结构，提出必要修改并直接完成改造、验证和总结
```

预期自动链路：

1. `clarify`
2. `plan`
3. `implement`
4. `review`
5. `verify`
6. `summary`

### 4.6 单 agent 先调研

```text
/ucc-flow-single-research 评估当前框架的模块边界、构建组织和测试缺口，并给出下一步落地建议
```

完成调研后，如果要继续做，就再显式触发：

```text
/ucc-flow-single-dev 基于刚才的结论继续落地修改
```

## 5. 如何判断当前是否要继续输入命令

这是很多团队误解最重的一点。判断规则可以简化成一句话：

只要流程还是 `running`，就不要继续手工输入下一阶段命令。

### 5.1 什么时候不用输入

以下情况都不需要你继续输入下一阶段命令：

1. 刚刚启动了 `/ucc-flow-team-*` 或 `/ucc-flow-single-*`
2. 当前输出里明确说明会自动推进下一节点
3. 当前状态仍是 `running`

### 5.2 什么时候才需要输入

只在以下情况需要显式继续：

1. 状态变成 `paused`
2. 输出里明确要求你补充信息
3. 验证失败，需要你确认处理方向
4. 存在高风险动作被 pausePolicy 拦住

此时统一使用：

```text
/ucc-flow-continue
```

如需先确认现状：

```text
/ucc-flow-status
```

如需结束当前 run：

```text
/ucc-flow-abort
```

## 6. 推荐的生产环境优化方向

如果你的目标是“尽可能少人工干预，但仍可控”，推荐保留下面这套模型。

### 6.1 公开命令面收敛

对团队培训只强调这 12 个命令：

1. `/ucc-flow-team-standard`
2. `/ucc-flow-team-fast`
3. `/ucc-flow-team-strict`
4. `/ucc-flow-team-review`
5. `/ucc-flow-team-research`
6. `/ucc-flow-team-doc`
7. `/ucc-flow-single-dev`
8. `/ucc-flow-single-review`
9. `/ucc-flow-single-research`
10. `/ucc-flow-status`
11. `/ucc-flow-continue`
12. `/ucc-flow-abort`

其余命令保留，但作为专业补充能力，不做团队主培训入口。

### 6.2 默认自动推进

建议维持：

- `executionModeDefault = auto`
- `team.standard` 使用 `balanced`
- `team.fast` 使用 `auto`
- `team.strict` 使用 `strict`

这套默认值已经覆盖大部分生产场景。

### 6.3 文档和校验要跟命令面同步

任何以下变更都必须同步文档和测试：

1. 新增或删除公开命令
2. 修改 workflow profile
3. 修改 pausePolicy
4. 修改 team / single 的自动接力逻辑

最小验证：

```bash
node scripts/validate-config.js
node tests/run-all.js
```

### 6.4 旧入口不要再公开推荐

历史入口可以保留在 `legacy/commands/` 做兼容参考，但不要继续在 README、使用说明、CLAUDE 或团队培训文档中推荐。

## 7. 如何继续定制

### 7.1 增加一个新的显式 flow 入口

需要同时改四处：

1. `commands/` 新增公开命令文件
2. `workflows/definitions.json` 新增 profile 或节点定义
3. 文档更新
4. 测试更新

命令 frontmatter 建议保持：

```yaml
---
description: 自定义流程入口
context: fork
agent: workflow-orchestrator
workflowCapable: true
workflowProfile: your.profile
workflowNode: your-entry
executionMode: auto
pausePolicy: balanced
triggerVisibility: always
---
```

### 7.2 如果还想继续减少命令数量

推荐的顺序不是继续拆更多 flow 命令，而是：

1. 保留显式完整 flow 入口
2. 控制公开培训材料只使用那 12 个主命令
3. 将其余命令视为补充工具
4. 让 runtime 内部根据条件决定调用哪个 agent

这比继续增加“阶段命令”更稳定，也更不容易误导团队成员。

## 8. 附录：面向你当前场景的直接示例

### 示例 1：团队模式，一次触发自动全流程

```text
/ucc-flow-team-standard 基于当前项目框架执行自动化治理：检查现状、提出必要修改、实施修改、完成验证、收尾并更新文档
```

### 示例 2：高风险改造

```text
/ucc-flow-team-strict 对当前认证、权限和配置加载链路做风险评估、重构、完整验证和文档同步
```

### 示例 3：先研究再落地

```text
/ucc-flow-team-research 研究当前框架的模块划分、构建结构和测试组织，如结论清晰则自动接力到标准交付流程
```

### 示例 4：单人自动闭环

```text
/ucc-flow-single-dev 基于当前仓库完成自动检查、必要修改、验证和总结
```

### 示例 5：暂停后的统一恢复

```text
/ucc-flow-status
/ucc-flow-continue
```

---

如果你要做的是“给团队一个一眼就看懂、还能在生产中稳定使用的自动化套路”，当前这套显式命令面加统一 runtime 的方向是合理的，而且比参数化入口和阶段命令更适合长期维护。
