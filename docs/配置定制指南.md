# UCC 配置定制指南

> 面向生产场景的显式命令面与自动接力 workflow 定制手册

本文只面向维护 UCC 配置仓库的开发者，负责说明如何扩展命令、workflow、hooks 与校验逻辑。

**文档版本：** v4.7.0

---

## 1. 设计目标

这套配置的核心目标不是“提供很多分散命令”，而是：

1. 让用户通过极少数显式完整入口启动完整流程
2. 让 runtime 在 team / single 两种模式下自动接力执行
3. 将人工介入压缩到暂停、失败、冲突和必要确认这些时刻
4. 在生产场景下保留风险控制、状态追踪和恢复能力

---

## 2. 当前公开命令面

### 2.1 目录快照

```text
agents/                 # 代理配置（20个）
commands/               # 斜杠命令（8个）
skills/                 # 技能模块（19个）
docs/                   # 维护文档（源仓库保留）
tests/                  # 配置测试（源仓库保留）
workflows/              # workflow 定义与运行时状态
```

### 2.2 主流程入口

B 方案公开给用户的 workflow 入口只有 5 个：

- `/ucc-team-standard`
- `/ucc-team-strict`
- `/ucc-team-research`
- `/ucc-single-standard`
- `/ucc-single-research`

控制命令只有 3 个：

- `/ucc-flow-status`
- `/ucc-flow-continue`
- `/ucc-flow-abort`

这 8 个命令就是完整的公开命令面。

低频专项能力不再单独占用 slash 命令，包括构建修复、语言专项审查、数据库审查、E2E、设计文档、交付文档、覆盖率补齐和死代码清理。这些能力改为由 runtime 自动调度内部 agent。

当前最小版并行化不改变 root workflow 的串行推进语义；在 `team.standard.plan`、`team.strict.detailed-plan`、`review` 节点内开放受控并行，并在 `team.standard.verify` 与 `team.strict.full-verify` 节点内开放有限并行验证，由 `team-orchestrator` 汇总结果后继续推进。运行时还会写入 `workflows/control/` 控制面快照，供状态命令和恢复场景读取。

默认部署包不会复制 `docs/`、`tests/` 和 `contexts/`。其中 `docs/` 与 `tests/` 只保留在配置源仓库中，`contexts/` 已退出当前结构。

当前默认安装模型也已经调整为“只复制 `.claude/` 下的运行时资产”。也就是说，执行 `node scripts/copy-config.js <target>` 时，默认不会生成目标项目根目录 `CLAUDE.md`，也不会默认生成 `.claude/settings.json` 或项目根 `.mcp.json`。如果需要显式接入 `CLAUDE`、hooks 或项目级 MCP，应分别使用：

- `--claude-mode dotclaude`
- `--claude-mode import-root`
- `--hooks local`
- `--hooks project`
- `--mcp project`
- 默认直接运行脚本，通过向导选择 hooks、MCP、平台和必要参数
- `--legacy-layout`（兼容旧布局）

### 2.3 为什么这样收敛

旧式阶段命令和参数化入口容易带来这些问题：

1. 用户不知道当前该输入哪个下一阶段命令
2. 看不出当前是 team 模式还是 single 模式
3. 命令越多，误用和培训成本越高
4. 很多人会把“自动接力”误解成“每阶段都得再输一次命令”

显式完整入口的好处是：

- `/ucc-team-standard` 一眼就知道是团队标准交付
- `/ucc-single-standard` 一眼就知道是单 agent 闭环
- `/ucc-flow-continue` 一眼就知道是恢复暂停流程

## 3. 自动接力运行模型

### 3.1 统一规则

所有公开 flow 入口都满足：

- `workflowCapable: true`
- `executionMode: auto`
- `triggerVisibility: always`

也就是说，用户触发入口后，runtime 默认自动推进，不需要每一阶段都人工补命令。

所有自定义 agents 默认 `model: inherit`，继承当前会话模型。

### 3.2 内部 profile

当前内部 profile 只保留：

- `team.standard`
- `team.strict`
- `team.research`
- `single.standard`
- `single.research`

review、verify、docs、quality-gate 仍然保留为内部节点，但不再作为用户记忆的公开主入口。

### 3.3 pausePolicy

当前 runtime 使用 `pausePolicy` 决定何时暂停：

- `auto`：最少打断，适合单 agent 标准闭环
- `balanced`：默认平衡，适合常规生产交付
- `strict`：更积极暂停，适合高风险场景

运行时只在以下情况暂停：

1. 命中对应 pausePolicy 的风险信号
2. 关键执行失败
3. 上下文冲突
4. 缺少不能合理推断的关键输入

## 4. 推荐用法与自动链路

### 4.1 团队标准交付

```text
/ucc-team-standard 审查当前项目框架，识别结构问题并给出最小可执行修改；完成修改、验证、收尾并同步文档
```

自动链路：

1. `clarify`
2. `plan`
3. `implement`
4. `review`
5. `verify`（节点内有限并行验证）
6. `docs`
7. `summary`

说明：`plan` 节点会并行生成实施计划，并在命中架构类风险信号时按需并行调度 `architect`；`review` 节点会并行审查；`verify` 节点在命中 `db-migration` 时可按需并行调度 `database-reviewer`。

### 4.2 团队严格交付

```text
/ucc-team-strict 审查并重构权限、配置加载与发布链路，要求完整验证、质量门禁和文档同步
```

自动链路：

1. `clarify`
2. `risk`
3. `detailed-plan`
4. `implement`
5. `review`
6. `full-verify`（节点内有限并行验证）
7. `docs`
8. `quality-gate`
9. `summary`

说明：`detailed-plan` 节点会并行生成详细计划，并在命中架构类风险信号时按需并行调度 `architect`；`review` 节点会并行审查；`full-verify` 节点在命中 `db-migration` 时可按需并行调度 `database-reviewer`。

### 4.3 团队调研并自动交接

```text
/ucc-team-research 评估当前项目框架的模块划分、构建结构和测试组织；若结论明确，则继续落地改造
```

自动链路：

1. `define-problem`
2. `evidence`
3. `conclusion`
4. `handoff`
5. 自动切入 `team.standard.plan`
6. 继续执行实施、审查、验证、文档同步和总结

### 4.4 单 agent 标准闭环

```text
/ucc-single-standard 基于当前仓库完成自动检查、必要修改、验证和总结
```

自动链路：

1. `clarify`
2. `plan`
3. `implement`
4. `review`
5. `verify`
6. `summary`

### 4.5 单 agent 调研并自动交接

```text
/ucc-single-research 分析当前仓库的模块边界、构建组织和测试缺口，并在结论明确后继续落地
```

自动链路：

1. `define-problem`
2. `evidence`
3. `conclusion`
4. `next-action`
5. 自动切入 `single.standard.plan`
6. 继续执行实施、审查、验证和总结

## 5. 团队成员该如何判断是否还要输入命令

判断原则很简单：

**只要当前 workflow 状态还是 `running`，就不要继续输入下一阶段命令。**

真正需要人工输入时，只会用到：

```text
/ucc-flow-status
/ucc-flow-continue
/ucc-flow-abort
```

这就是生产环境下更稳定的交互模型：入口少，接力自动，控制统一。

## 6. 如何修改命令面

### 6.1 新增一个新的显式 flow 入口

需要同步修改四处：

1. `commands/` 新增公开命令文件
2. `workflows/definitions.json` 新增 profile 或节点定义
3. 文档更新
4. 测试更新

建议 frontmatter 保持：

```yaml
---
description: 自定义流程入口
context: fork
agent: workflow-orchestrator
workflowCapable: true
workflowProfile: your.profile
workflowNode: your-entry
executionMode: auto
pausePolicy: balanced
triggerVisibility: always
---
```

### 6.2 不要再新增阶段公开命令或低频专项命令

推荐顺序不是再加“计划命令”“开发命令”“验证命令”，而是：

1. 保留显式完整 flow 入口
2. 将 review / verify / docs 以及各种专项能力保持为内部节点或内部 agent
3. 通过 runtime 根据上下文决定调用哪个 agent
4. 只让用户在暂停时使用统一控制命令

这比继续扩展阶段命令更稳定，也更适合长期维护。

---

## 7. 配置变更后的最小验证

每次修改命令面、workflow、代理或文档后，至少执行：

```bash
node scripts/validate-config.js
node tests/run-all.js
```

当前 `validate-config.js` 的最小合同已经不仅是“文件存在”，还包括：

1. workflow profile / node / transition 的语义闭环
2. `hooks/hooks.json` 中 `PreToolUse` / `PostToolUse` / `Stop` 的结构完整性
3. docs / hooks / rules / skills 中退役 `/ucc-*` 命令引用的扫描
4. `docs/配置定制指南.md` 与 README 版本快照一致性

如果新增或删除公开命令，还要同时检查：

1. `commands/` 数量快照
2. `docs/配置定制指南.md` 中的目录快照
3. README 版本日志
4. workflow runtime 相关测试

## 8. 面向当前场景的直接示例

### 示例 1：团队模式，一次触发自动全流程

```text
/ucc-team-standard 基于当前项目框架执行自动化治理：检查现状、提出必要修改、实施修改、完成验证、收尾并更新文档
```

### 示例 2：高风险改造

```text
/ucc-team-strict 对当前认证、权限和配置加载链路做风险评估、重构、完整验证和文档同步
```

### 示例 3：先研究再落地

```text
/ucc-team-research 研究当前框架的模块划分、构建结构和测试组织，如结论清晰则自动接力到标准交付流程
```

### 示例 4：单人自动闭环

```text
/ucc-single-standard 基于当前仓库完成自动检查、必要修改、验证和总结
```

### 示例 5：暂停后的统一恢复

```text
/ucc-flow-status
/ucc-flow-continue
```

如果目标是“给团队一套一眼就看懂、还能在生产中稳定使用的自动化套路”，当前这套 B 方案就是推荐的默认答案。
