## 一、这是什么

**UCC（Unified Claude Code Config）** 是一套 Claude Code 配置插件，部署后能让 Claude Code 具备：

- 一致的编码规范（Go / TypeScript / Vue / JavaScript / Node.js）
- 专业代理（Agent）处理特定任务
- 显式完整命令触发标准化工作流
- 自动推进 runtime，在需要时再暂停人工介入
- 可选的 Hooks 自动拦截危险操作

## 二、如何部署

### 方式 A：项目级部署（推荐团队使用）

Claude Code 的项目级入口应为：

- 项目根目录的 `CLAUDE.md`
- 项目根目录下 `.claude/commands`
- 项目根目录下 `.claude/agents`
- 项目根目录下 `.claude/settings.json`

推荐直接使用复制脚本：

```bash
# 在配置仓库中执行
node scripts/copy-config.js /path/to/your-project
```

复制完成后，目标项目结构应为：

```text
your-project/
├── CLAUDE.md
└── .claude/
    ├── agents/
    ├── commands/
    ├── skills/
    ├── scripts/
    ├── workflows/
    └── settings.json
```

### 方式 B：手动复制

```bash
cp CLAUDE.md /path/to/your-project/
cp -r README.md rules agents commands skills contexts hooks scripts mcp-configs docs tests workflows /path/to/your-project/.claude/
cp hooks/project-settings.json /path/to/your-project/.claude/settings.json
```

### 方式 C：全局部署（仅高级用法）

全局部署适合个人统一偏好，但更容易与其他配置混用。若使用全局部署，请自行确认 `CLAUDE.md`、命令和 settings 的优先级关系。

## 三、命名空间与验证

每次对话中，可通过以下方式确认「UCC 配置已生效」：

```bash
# 推荐：直接使用显式完整入口
/ucc-flow-team-standard 实现用户登录功能

# 快速模式
/ucc-flow-team-fast 修复登录页禁用状态

# 单人开发闭环
/ucc-flow-single-dev 整理当前模块边界并补齐验证

# 兜底：在消息中加 @ucc 标记
@ucc 帮我写一个 Go HTTP handler
```

**生效标志**：输出末尾出现 `配置标识：UCC`

如果没有出现，说明 Claude Code 没有加载到本配置。优先检查：

1. 项目根目录是否有 `CLAUDE.md`
2. 项目内是否有 `.claude/settings.json`
3. `.claude/commands/` 和 `.claude/agents/` 是否完整复制
4. `.claude/workflows/definitions.json` 是否已复制

## 五、代理（Agent）使用说明

代理是专业化的子任务处理器。公开入口只暴露 flow 命令，具体 agent 由 runtime 根据节点和改动内容自动选择。

### 5.1 通过斜杠命令隐式触发（推荐）

```bash
/ucc-flow-team-standard  → 自动调用 team-orchestrator 编排团队节点
/ucc-flow-team-strict    → 自动调用 team-orchestrator + 严格 pausePolicy
/ucc-flow-single-dev     → 自动调用 workflow-orchestrator 编排单人节点
/ucc-flow-status         → 自动调用 workflow-orchestrator 查询状态
/ucc-go-review           → 自动调用 go-reviewer
```

### 5.2 直接在对话中请求

```text
帮我用 planner 代理规划这个功能
用 security-reviewer 检查这段代码
```

### 5.3 代理自动触发规则（Claude 会主动调用）

| 触发条件 | 自动调用的代理 |
|---------|---------------|
| 使用 `/ucc-flow-team-*` | `team-orchestrator` |
| 使用 `/ucc-flow-single-*` | `workflow-orchestrator` |
| 代码写完 | `code-reviewer` 或语言专项 reviewer |
| 含敏感操作的代码 | `security-reviewer` |
| 构建/类型检查失败 | `build-error-resolver` |
| 文档/接口变更 | `doc-updater` |
| 复杂功能请求 | `planner` |
| 数据库变更 | `database-reviewer` |

## 六、工作模式（Contexts）说明

三种模式影响 Claude 的行为风格：

| 模式 | 行为特点 | 适用场景 |
|------|---------|----------|
| **开发模式** (`dev`) | 先写代码，后解释；优先可工作方案 | 活跃编码期 |
| **审查模式** (`review`) | 先读懂代码，再评论；输出发现报告 | PR 审查 |
| **研究模式** (`research`) | 先理解，后行动；输出调研结论 | 探索新技术 |

切换命令：`/ucc-context-dev` / `/ucc-context-review` / `/ucc-context-research`

## 七、Hooks 激活说明（重要）

Hooks 是**硬性拦截**机制。使用 `scripts/copy-config.js` 做项目级部署时，会自动生成项目内的 `.claude/settings.json`；手动部署时，需要从 `hooks/project-settings.json` 复制到项目的 `.claude/settings.json`。

### 7.1 五个 Hook 的作用

| Hook | 触发时机 | 拦截内容 |
|------|---------|----------|
| `pre:cmd:risk-blocker` | AI 执行 Shell 命令前 | `rm -rf`、危险格式化命令 |
| `pre:write:sensitive-check` | AI 写文件前 | 硬编码密码、API Key、Token |
| `post:write:ts-check` | 写入 TS/TSX/Vue 文件后 | 自动运行 `tsc --noEmit` + ESLint |
| `post:write:go-check` | 写入 `.go` 文件后 | 自动运行 `go vet` + `gofmt` 检查（`staticcheck` 可选） |
| `stop:delivery-reminder` | 每次响应完成后 | 输出 workflow 摘要和最小交付提醒（不阻断） |

## 九、典型工作流示例

### 对已有项目框架做完整治理

```text
1. /ucc-flow-team-standard 审查现状，提出必要修改，实施修改，执行验证，收尾并同步文档
   -> runtime 创建 team.standard run
   -> 自动推进 clarify -> plan -> implement -> review -> verify -> docs -> summary

2. 只有在命中 pausePolicy、危险改动、失败或冲突时才暂停
   -> 使用 /ucc-flow-status 查看状态
   -> 使用 /ucc-flow-continue 继续

3. 如任务很小，可改用 /ucc-flow-team-fast
4. 如任务高风险，可改用 /ucc-flow-team-strict
```

### 团队 PR 审查

```text
1. /ucc-flow-team-review
2. runtime 自动推进 collect-context -> review -> risk-gap -> summary
3. 如需专项审查，可补充 /ucc-go-review 或 /ucc-typescript-review
```

### 单人调研并继续落地

```text
1. /ucc-flow-single-research
2. runtime 自动推进 define-problem -> evidence -> conclusion -> next-action
3. 需要继续改造时，直接使用 /ucc-flow-single-dev
```

## 十、配置定制入口

详细定制方法参见 `docs/配置定制指南.md`，以下是常见定制场景：

### 10.1 添加新的编码规则

在对应语言目录下新建 `.md` 文件。

### 10.2 添加新的显式流程命令

在 `commands/` 下新建 `.md` 文件，并补齐 workflow frontmatter：

```yaml
---
description: 我的流程命令
context: fork
agent: workflow-orchestrator
workflowCapable: true
workflowProfile: my.profile
workflowNode: my-entry-node
executionMode: auto
pausePolicy: balanced
triggerVisibility: always
---
```

### 10.3 在项目中覆盖全局规则

在项目根目录的 `CLAUDE.md` 中追加覆盖说明，项目级规则优先级高于全局。

### 10.4 关闭某个 Hook

项目级部署时，编辑项目内的 `.claude/settings.json`，删除对应的 hook 条目即可；如果你使用的是全局部署，再编辑 `~/.claude/settings.json`。
