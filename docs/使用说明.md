## 一、这是什么

**UCC（Unified Claude Code Config）** 是一套 Claude Code 配置插件，部署后可以提供：

- 一致的编码规范（Go / TypeScript / Vue / JavaScript / Node.js）
- 专业代理（Agent）处理特定任务
- 显式完整命令触发标准化工作流
- 自动推进 runtime，在需要时再暂停人工介入
- 可选 Hooks 拦截危险操作

核心思想不是“给很多阶段命令”，而是“用少量可读入口启动完整自动化流程”。

## 二、如何部署

### 方式 A：项目级部署（推荐）

Claude Code 的项目级入口应为：

- 项目根目录的 `CLAUDE.md`
- 项目根目录下 `.claude/commands`
- 项目根目录下 `.claude/agents`
- 项目根目录下 `.claude/settings.json`

推荐使用复制脚本：

```bash
node scripts/copy-config.js /path/to/your-project
```

复制完成后，目标项目结构应为：

```text
your-project/
├── CLAUDE.md
└── .claude/
    ├── agents/
    ├── commands/
    ├── skills/
    ├── scripts/
    ├── workflows/
    └── settings.json
```

### 方式 B：手动复制

```bash
cp CLAUDE.md /path/to/your-project/
cp -r README.md rules agents commands skills contexts hooks scripts mcp-configs docs tests workflows /path/to/your-project/.claude/
cp hooks/project-settings.json /path/to/your-project/.claude/settings.json
```

## 三、命名空间与验证

推荐直接用显式完整入口：

```text
/ucc-team-standard 审查当前项目框架并完成必要改造、验证、文档同步和总结
/ucc-single-standard 基于当前仓库完成自动检查、修改、验证和总结
@ucc 帮我分析这段 TypeScript 代码
```

**生效标志**：输出末尾出现 `配置标识：UCC`

如果没有出现，优先检查：

1. 项目根目录是否有 `CLAUDE.md`
2. 项目内是否有 `.claude/settings.json`
3. `.claude/commands/` 和 `.claude/agents/` 是否完整复制
4. `.claude/workflows/definitions.json` 是否已复制

## 四、公开命令面（25个）

### 4.1 自动化流程入口

| 命令 | 作用 | 自动链路 |
|------|------|----------|
| `/ucc-team-standard` | 标准团队交付闭环 | 澄清 -> 计划 -> 实施 -> 审查 -> 验证 -> 文档 -> 总结 |
| `/ucc-team-strict` | 严格团队交付闭环 | 澄清 -> 风险 -> 详细计划 -> 实施 -> 审查 -> 完整验证 -> 文档 -> 质量门禁 -> 总结 |
| `/ucc-team-research` | 团队调研闭环 | 问题定义 -> 证据 -> 结论 -> 交接 -> 自动切入 `team.standard.plan` |
| `/ucc-single-standard` | 单 agent 标准闭环 | 澄清 -> 计划 -> 实施 -> 审查 -> 验证 -> 总结 |
| `/ucc-single-research` | 单 agent 调研闭环 | 问题定义 -> 证据 -> 结论 -> 后续动作 -> 自动切入 `single.standard.plan` |

### 4.2 流程控制命令

| 命令 | 作用 |
|------|------|
| `/ucc-flow-status` | 查看当前或指定 workflow run 的状态 |
| `/ucc-flow-continue` | 继续当前或指定的 paused run |
| `/ucc-flow-abort` | 中止当前或指定的 workflow run |

## 五、B 方案自动化流程

这部分回答一个核心问题：**用户触发命令后，会不会按照需求自动走完全流程？**

答案是：**会**。前提是使用的是上面 5 个显式 flow 入口之一。

### 5.1 `/ucc-team-standard`

适用场景：

- 已有项目框架雏形
- 希望自动检查、提出修改、实施修改、验证、收尾、更新文档
- 希望尽量少人工干预

输入方式：

```text
/ucc-team-standard 审查当前项目框架，识别结构问题并给出最小可执行修改；完成修改、验证、收尾并同步文档
```

系统预期流程：

1. `clarify`：澄清目标、边界、限制
2. `plan`：输出最小可执行计划
3. `implement`：实施改动
4. `review`：检查风险、回归和遗漏
5. `verify`：执行验证
6. `docs`：同步必要文档
7. `summary`：输出交付总结

用户在正常情况下不需要继续输入下一阶段命令。

### 5.2 `/ucc-team-strict`

适用场景：

- 核心模块
- 高风险改动
- 需要完整质量门禁

输入方式：

```text
/ucc-team-strict 审查并重构权限、配置加载与发布链路，要求完整验证、质量门禁和文档同步
```

系统预期流程：

1. `clarify`
2. `risk`
3. `detailed-plan`
4. `implement`
5. `review`
6. `full-verify`
7. `docs`
8. `quality-gate`
9. `summary`

### 5.3 `/ucc-team-research`

适用场景：

- 先研究现状
- 需要证据支持的改造建议
- 希望研究结束后自动衔接实施

输入方式：

```text
/ucc-team-research 评估当前项目框架的模块划分、构建结构和测试组织；若结论明确，则继续落地改造
```

系统预期流程：

1. `define-problem`
2. `evidence`
3. `conclusion`
4. `handoff`
5. 自动切入 `team.standard.plan`
6. 继续执行实施、审查、验证、文档同步和总结

这就是 team 模式下“研究后自动接力到交付”的标准套路。

### 5.4 `/ucc-single-standard`

适用场景：

- 单人维护
- 日常修复
- 中小型重构

输入方式：

```text
/ucc-single-standard 基于当前仓库完成自动检查、必要修改、验证和总结
```

系统预期流程：

1. `clarify`
2. `plan`
3. `implement`
4. `review`
5. `verify`
6. `summary`

### 5.5 `/ucc-single-research`

适用场景：

- 单人调研
- 需要先分析问题和证据
- 希望在可执行时自动继续落地

输入方式：

```text
/ucc-single-research 分析当前仓库的模块边界、构建组织和测试缺口，并在结论明确后继续落地
```

系统预期流程：

1. `define-problem`
2. `evidence`
3. `conclusion`
4. `next-action`
5. 自动切入 `single.standard.plan`
6. 继续执行实施、审查、验证和总结

## 六、何时需要继续输入命令

判断规则可以简化成一句话：

**只要流程还是 `running`，就不要继续手工输入下一阶段命令。**

### 6.1 不需要继续输入的情况

1. 刚刚启动了 `/ucc-team-*` 或 `/ucc-single-*`
2. 输出里明确说明会自动推进下一节点
3. 当前状态仍是 `running`

### 6.2 需要继续输入的情况

只在以下情况才需要人工接力：

1. 状态变成 `paused`
2. 输出里明确要求补充信息
3. 验证失败，需要确认处理方向
4. 存在高风险动作被 pausePolicy 拦住

此时统一使用：

```text
/ucc-flow-status
/ucc-flow-continue
```

如果需要终止：

```text
/ucc-flow-abort
```

## 七、专项命令与模式命令

专项能力命令不是主流程入口，而是补充工具。团队培训时，优先只讲 8 个 workflow 相关命令：

1. `/ucc-team-standard`
2. `/ucc-team-strict`
3. `/ucc-team-research`
4. `/ucc-single-standard`
5. `/ucc-single-research`
6. `/ucc-flow-status`
7. `/ucc-flow-continue`
8. `/ucc-flow-abort`

其余命令保留给专项场景，例如：

- `/ucc-go-review`
- `/ucc-db-review`
- `/ucc-design-doc`
- `/ucc-delivery-doc`
- `/ucc-e2e`
- `/ucc-test-coverage`

所有自定义 agent 默认 `model: inherit`，继承当前会话模型，不额外绑定固定官方模型别名。

## 八、生产环境建议

### 8.1 默认推荐

- 常规任务：`/ucc-team-standard`
- 高风险任务：`/ucc-team-strict`
- 先研究再落地：`/ucc-team-research`
- 单人闭环：`/ucc-single-standard`

### 8.2 不建议的做法

- 不要把每个阶段再拆成用户手工命令
- 不要让团队成员去记 profile、node、pausePolicy 参数
- 不要公开推荐已经移除的旧入口

### 8.3 配置修改后的最小验证

```bash
node scripts/validate-config.js
node tests/run-all.js
```

如果你的目标是“让团队成员一眼看懂，并且一条命令就能在生产场景中走完整自动化流程”，当前这套 B 方案就是为这个目标设计的。
